# Codex Feature Gap Tracker

This tracker keeps the Codex-inspired implementation honest while Godex evolves.

## Implemented

- Godot editor main screen named `Godex`.
- Codex-like left navigation, central conversation, right-aligned user prompt bubbles, floating right progress/output/capability overlay, and bottom composer.
- OpenAI Responses API and Chat Completions compatible request payload builders.
- Context compression threshold, message-tail retention, manual and automatic session-compaction records, Automation summaries, capability-preview before/after token estimates, context-window warning, bounded compaction history rows, and composer add-context/send affordances for low-context warnings.
- Session-backed sub-agent task records for parallel investigation drafts, branch/read-only metadata, lifecycle status, timestamps, summaries/results/errors, cancellation metadata, result handoff metadata, worker notifications, local child-session records, parent-child edge topology, right-inspector rows, Automation summaries/actions, and restore coverage.
- MCP endpoint setting and visible connection status.
- Provider, API mode, base URL, model, API key/env, MCP, Skill, and context-compression settings with JSON persistence.
- Local Skill registry and Settings manager for scanning `SKILL.md` roots, parsing `agents/openai.yaml` interface/dependency/policy metadata, searching local Skills, persisting disabled paths, showing enabled Skill capability hints, and injecting explicitly mentioned `$skill` instructions for a single Agent turn.
- Non-default Yuren OpenAI-compatible provider metadata, model limits, attachment/reasoning variants, supported model choices (`gpt-5.4-mini`, `gpt-5.4`, `gpt-5.5`), and startup migration for explicit development validation settings.
- OpenAI API runtime configuration snapshot with endpoint resolution, key source detection, masked display, and request headers.
- OpenAI execution service boundary for request readiness, text parsing, tool-call extraction, and normalized API errors.
- Session-scoped model events with redacted OpenAI request metadata and Automation view summaries.
- MCP tool-call dispatch readiness with approval gating, Godex context mapping, and dispatch audit events.
- First-class local MCP probe events that turn the Automation probe action into auditable model events plus normal pending tool-call state whose invoked MCP provider appears in source chips.
- MCP tools/list discovery request parsing, cached tool summaries, and MCP management view status.
- Explicit OpenAI tool-result continuation queue with Automation view status and a separate send action.
- Automation continuation preview for endpoint, model, API mode, key source, status, and external-send risk.
- State-aware Automation send-button gating for pending OpenAI continuation requests.
- Composer stop action for canceling in-flight OpenAI requests and recording canceled transport events.
- Composer retry action for failed or canceled OpenAI requests, with in-memory raw transport retention and redacted UI summaries.
- Bounded Agent loop state and automatic MCP tool-call advancement after model tool-call responses, stopping on approval, cancellation, failed requests, missing credentials, pending manual continuation, or max-step guard.
- Review-mode OpenAI send approval gates for first-turn, retry, and manual tool-result continuation requests, with source metadata and approved-dispatch replay.
- Godex header refresh action for safely rebuilding the editor main screen after addon updates.
- Local headless smoke test and CI workflow skeleton.
- Real center-view switching for Search, Plugins, Automation, and Settings navigation entries. MCP remains a settings and `/mcp` command surface, not a top-level plugin navigation item.
- Maintained local Codex screenshot reference index under `docs/references/`.
- Persistent session store for active thread restore, message history, search records, archive metadata, and fork metadata.
- Local slash command handling for basic session control, session-backed active goal records, and manual compaction without starting an OpenAI request.
- Discoverable composer slash-command suggestion panel with prefix filtering, alias matching, parameter hints, a Codex-style floating action-list popup above the input, selected-row highlight, click-to-insert behavior, and keyboard Up/Down, Enter, and Esc handling.
- Compact Codex-style composer surface with a dark multiline input, fixed bottom action row, a scene-owned `+` add-context menu for safe local context actions, selected-text reference chips, reference-only send support, and first-stage image URL/data reference serialization for model requests.
- State-backed composer Plan Mode toggle that persists with settings, records model-event audit data, injects a Codex Plan Mode instruction contract, and suppresses OpenAI tool schemas for the next initial request.
- Codex-style direct composer sending with running-turn queued user-message rows, first-stage idle queue draining, queued slash-command parsing through the local command boundary, queued `!` shell prompt handoff into command-run approval records, and next-turn guide instructions that are preserved in Automation summaries, one-turn Agent instructions, submitted-turn audit records, and per-row guide/delete actions.
- Session-backed goal tracking that stores objective/status/timing metadata on the active thread, restores with session history, records model events, exposes Automation summaries, and injects active goals into Agent instructions.
- First-stage Codex-style changed-file review strip above the composer, backed by persisted review-summary state, fixed-width addition/removal counters, expandable file rows, a read-only Git summary parser/tool publication entry, and a review action placeholder.
- Floating add-context, approval, slash-command, model, and reasoning composer popovers under `ComposerPopoverLayer` that do not resize the input surface when opened, with the approval menu constrained to a short fixed content height to avoid the previous 525px blank panel and bottom overflow.
- Codex-style combined composer model/reasoning picker with compact checked rows, a hover-only right-side model submenu, catalog model choices, custom-model visibility, and state updates that preserve the existing OpenAI request pipeline.
- Provider-catalog-backed default-model and API-mode selection in settings, so supported OpenAI-compatible model names are visible, invalid persisted Yuren model names normalize before a request is built, and stale Responses API UI state cannot override Yuren's Chat Completions preset.
- OpenAI stream HTTP-error diagnostics with bounded provider body previews, stage-aware progress/audit labels, runtime provider normalization before first-turn, continuation, and retry dispatch, a direct non-stream first-send path for explicit OpenAI-compatible Chat Completions providers, a one-shot non-stream fallback for stream/SSE compatibility failures, and a final plain Chat Completions compatibility fallback that recomputes `/v1/chat/completions`, strips streaming/tool/reasoning-only fields, sanitizes tool-result continuation messages into plain text, treats empty completions as retryable failures rather than placeholder success, keeps the same assistant message and retry metadata, and hides successful transport bookkeeping from the chat transcript.

- Floating single-surface right progress overlay under `ProgressOverlayLayer`, with compact sections instead of a layout-reserving three-card right rail.
- Conversation information stream rows for Agent loop, MCP context, sub-agent, compaction, plan, and goal state, rebuilt from state-level records instead of static progress/output placeholders.
- OpenAI request-build, transport, response, and raw stream-trace metadata remain rebuildable in model-event/progress audit surfaces without becoming chat transcript rows.
- Local Responses-compatible model replay that covers deterministic assistant text, MCP tool-call parsing, pending tool-call creation, and local-replay audit labeling without requiring an API key or sending an external request.
- Event-driven right inspector sections for progress, generated artifacts, sub-agents, and invoked external-tool sources, with empty states for new conversations and hidden sub-agent/progress sections until real events exist.
- Codex-aligned right-inspector semantics where `输出` lists artifacts or changed files and `来源` lists invoked external tools such as Godot .NET MCP, not OpenAI transport audits, provider probes, local replay fixtures, file/context attachments, or other local bookkeeping events.
- State-owned right-inspector output artifacts, with changed-file review summaries persisted as file artifacts, save/restore coverage, source/path deduplication, and review-clear cleanup that preserves unrelated manual artifacts.
- Codex-style settings workspace skeleton with a dedicated settings rail, `返回应用` row, centered scrollable settings column, grouped setting cards, row-aligned controls, real category filtering, settings search filtering with no-results feedback, editor-theme category icons, disabled unsupported appearance settings, and chat-composer/right-inspector hiding while settings mode is active.
- Transcript disclosure safety for tool-call arguments, using safe-key summaries and redaction instead of dumping long partial arguments or obvious API-key fragments.
- Explicit Godex main-screen activation fallback for editor automation and refresh validation, layered on top of the normal Godot main-screen plugin callbacks.
- Borderless assistant, tool, and command transcript rows, keeping only subtle user bubbles so the conversation reads like Codex's continuous chat surface.
- Top-right Codex-style layout chrome with a launcher menu button, a real bottom output drawer toggle, and the project/thread sidebar toggle without conflating those controls with MCP or plugin management.
- Codex-style bottom terminal drawer that uses the top-right bottom-panel toggle, preserves the fixed conversation width, and renders recent command-run audit status, runner metadata, output previews, and timelines from session events.
- Fixed-width centered transcript layout with restored rounded outer main chrome, stable transcript/composer/review widths, and transparent row padding so right overlays and bottom drawers do not squeeze chat text.
- Project-aware left rail labeling and empty-chat prompt text sourced from Godot `ProjectSettings` or MCP project-state data instead of a static placeholder.
- Full-width rounded sidebar conversation capsules for active and hovered rows, with hover-only inline pin/archive actions. Archive requires a second red confirmation click and resets to the initial age-only state when hover leaves the row.
- Settings-owned archived conversations page with archive-only search, explicit delete, and cancel-archive restore actions.
- Approval checkpoint records with pending approval display and approve/reject decision persistence.

## High Priority Gaps

- Richer streaming event replay with full persisted traces and inspectable argument panes. The current path keeps successful OpenAI request/transport/response bookkeeping out of the chat transcript, preserves those diagnostics in model events/progress surfaces, parses and accumulates first-stage Responses/Chat tool-call fragments, shows a lightweight `正在思考` shimmer while a request is in flight, keeps failures visible through the assistant error path, and uses transient `正在解析` partial argument preview rows that merge index/id deltas before final tool rows are recorded. It still lacks a replay UI for the complete event trace and editable/inspectable streamed tool-argument panes.
- Broader OpenAI conversation fixtures that can safely exercise streamed SSE text, streamed tool calls, direct non-stream Chat Completions first sends, stream-failure-to-non-stream fallback, plain Chat Completions compatibility fallback, Yuren-style Chat Completions delta closure, live MCP execution, and fuller end-to-end MCP execution coverage from the UI without requiring a real API key. The local MCP probe validates the Godot MCP leg, local model replay validates deterministic Responses parsing and pending tool-call creation, local tool-result continuation replay validates the ready continuation-to-final-assistant-message boundary, Yuren Chat continuation coverage protects the `/v1/chat/completions` second hop, and the Provider probe isolates live editor-process endpoint/model/auth/parser health from the larger streaming loop.
- Full composer queue and steer execution. The current send button dispatches immediately when idle and records queued user messages when a turn is running; guide instructions are submitted once during the next Agent request build; queued user messages can drain one at a time when the Agent loop is stopped, credentials are ready, and the composer has no unsent draft; queued slash commands can drain locally without OpenAI credentials; queued `!` shell prompts hand off to the existing command approval state machine instead of executing directly; and queued rows can be guided or cancelled from above the composer before they drain. It still does not interrupt a running turn to submit a live steer, preserve rejected steers ahead of queued user messages, or model Codex review-mode steer rejection semantics.
- Full Codex-style tool execution disclosure with command groups, live stdout/stderr panes, animated file-change counters, persisted collapse state, and replayable execution timelines. The current UI has default-collapsed MCP tool and command transcript rows with semantic tool/terminal icons, full-width hover-highlight disclosure headers, and independent per-use expansion, rebuilt from state-level transcript items; command rows now render structured runner/duration/exit/combined-output/stdout/stderr sections, bounded stdout/stderr chunk rows, and command status timelines with transcript-level redaction and truncation. Approved short commands can now run through a synchronous local `OS.execute` runner, but live streams and grouped command views are still missing.
- Broader execution-layer approval enforcement before file writes, runtime control, and every remaining high-risk MCP or network path not yet backed by concrete transport. Command requests now have a state-level approval-bound execution skeleton with blocked-command filtering, shell normalization, project-local working-directory checks, timeout-aware fingerprint validation, fake-runner smoke coverage, command-aware approval decisions, cancellation, single-running-command gating, Automation command approval summaries, command-run status rows, separate request/execute/cancel command buttons, transcript status updates, structured short-command audit metadata, and a first-stage approved short-command runner. The remaining gap is a proper process manager with process IDs, hard timeout enforcement, process-tree cancellation, separated stdout/stderr streaming, and background command sessions.
- Full session management UI for richer menu contents and deeper history workflows. Sidebar rows now expose hover-only Codex-style pin/archive actions, with archive guarded by a red second-click confirmation that resets on hover exit. Rename/fork remain secondary menu actions, pinned sessions are visually grouped above recent sessions without changing the session data shape, search results can reopen matching sessions, and archived conversations now live in Settings with search, delete, and cancel-archive restore actions. Remaining gaps include richer bulk history management and durable undo affordances.
- Full diff review and patch application workflow. The current changed-file strip is a summary/review-entry surface only; its Git summary service parses supplied read-only text/rows but does not run git, open a real diff pane, undo changes, or apply patches.
- Codex-native base tool protocols for model-visible command and patch actions plus the sandboxed filesystem backend. Godex currently exposes a first-stage Codex-named `exec_command` schema that bridges into the existing approval-bound command-run state machine, keeps `godex_command_request` as a compatibility alias, and reports `write_stdin` as explicitly unsupported until an interactive process manager exists. It still lacks full `shell_command`/`exec_command` process sessions, live `write_stdin`, `apply_patch` freeform handling, and Codex's internal exec-server-style sandboxed filesystem backend (`fs_read_file`/`fs_write_file` are backend protocol concepts, not primary model-visible base tools).
- Real parallel sub-agent orchestration with isolated sessions or worktrees and failure recovery. The current path records durable task cards, status updates, cancellation metadata, result handoff metadata, worker notifications, local child-session records, parent-child edge topology, right-inspector summaries, Automation summaries, and first-stage cancel/handoff buttons, but it does not launch external agents or create isolated execution contexts.
- Screenshot/context attachments for editor and runtime visual verification.
- Full Codex Skill lifecycle parity. Godex now has a local Skill registry, Settings manager, enabled Skill prompt hints, path-level disablement, and explicit `$skill` `SKILL.md` injection, but it still cannot install curated or GitHub Skills, create/scaffold new Skills, update local Skill packages, or browse/manage a marketplace-style Skill catalog.
- Complete local image input and richer reference attachments. The current reference model can carry selected text and URL/`data:image/` image references into the next request, but it still lacks a native image picker, clipboard/file-drop image capture, local file-to-data conversion, persistent thumbnails, editor/runtime screenshot attachment capture, selected Godot resource references, side-chat/browser/terminal context, and plugin-provided reference sources.
- Full add-context ingestion for file contents, photos, editor screenshots, selected Godot resources, side-chat/browser/terminal context, and later plugin-provided context sources. The current `+` menu exposes safe local actions: current-project MCP summary context, optional recommended changed-file paths, Plan Mode request-boundary toggling, session-backed goal tracking, and plugin navigation. Project-summary, file, Plan Mode, and goal rows now create auditable model/context events, and Plan Mode now suppresses initial request tool schemas, but Godex still does not read file contents, open a native file/photo picker, attach screenshots, or enable unsupported context-source rows.
- Full Codex Plan Mode workflow with an explicit proposed-plan rendering surface, confirmation/exit flow, and richer conversational mode affordances. The current implementation only enforces the initial OpenAI request boundary and audit trail.
- Multi-server MCP management with durable server IDs, add-server flow, tool allowlists, health checks, and transport-specific settings. The current settings workspace has a state-backed single `Godot .NET MCP` server row with endpoint editing, enable/disable, status, tool count, refresh, and edit affordances, but it is not yet the full MCP server manager shown in the Codex reference.
- Real appearance/theme preferences. The current settings rail deliberately disables `外观` until a durable theme preference and persistence model exists.
- Secure API key storage; the current editor setting supports environment variables and manual input but still needs a safer secret backend.
- Provider/model-aware reasoning availability, so unsupported effort levels can be disabled or hidden instead of showing the same four choices for every provider.

## Medium Priority Gaps

- Skills/workflow templates for common Godot tasks.
- Git branch/worktree creation from the UI.
- Rich validation result cards with logs and screenshots.
- Richer side-pane modes behind the top-right layout controls, including side chat, review/reference panes, terminal/browser launchers, and persisted panel widths.
- Dedicated view-transition animation for chat/search/plugins/settings surfaces. Width tweening is intentionally avoided for the shared conversation column because it can make text collide with overlays.

## Later Stage Gaps

- Codex-compatible plugin system and marketplace-style management UI. A plugin-system screenshot reference is preserved in `docs/references/codex-desktop-ux.md`, but this should wait until the core Godot Agent loop, MCP execution, approvals, sessions, and model/provider flows are mature.

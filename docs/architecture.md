# Godex Architecture

Godex is organized as a Godot editor main screen plugin. The top-level `EditorPlugin` only owns lifecycle and main-screen registration. UI state, agent orchestration, OpenAI payload construction, context compression, and sub-agent bookkeeping live in separate scripts under `addons/godex/core`.

## Layers

- `plugin.gd`: Registers the `Godex` main screen beside the built-in Godot editor screens through `EditorInterface.get_editor_main_screen()` and exposes an explicit activation fallback for editor automation.
- `ui/godex_main.tscn`: Godot-native Control tree mirroring the Codex desktop layout.
- `ui/godex_dock_controller.gd`: Binds the Control tree, paints editor theme chrome, and forwards user actions to services.
- `core/agent_service.gd`: Builds an auditable Agent turn from prompt, context, tools, approvals, and OpenAI payloads.
- `core/openai_request_builder.gd`: Produces Responses API and Chat Completions compatible payloads, endpoints, headers, and masked key labels.
- `core/openai_execution_service.gd`: Builds auditable request readiness snapshots and parses OpenAI-compatible responses, streaming deltas, completed response payloads, tool calls, and API errors.
- `core/provider_catalog.gd`: Stores auditable provider presets, model choices, reasoning variants, and OpenAI-compatible base URL defaults.
- `core/context_compressor.gd`: Keeps long sessions bounded by summarizing older messages.
- `core/subagent_manager.gd`: Creates local sub-agent task drafts, while `GodexState` stores the session-backed task records that UI and Automation can replay.
- `core/mcp_client.gd`: Builds auditable MCP context/tool requests for the configured project endpoint.
- `core/approval_policy.gd`: Classifies risky actions and produces approval checkpoints.
- `core/command_capability.gd`: Models command-line requests without enabling execution by default.
- `core/settings_store.gd`: Persists local user settings under `user://godex`.
- `core/session_store.gd`: Persists local conversation sessions, active thread selection, message history, archive metadata, and fork metadata under `user://godex`.

## Current Integration Boundary

Godex does not reimplement Godot editor tools. It treats the configured Godot .NET MCP endpoint as the project capability surface and exposes those capabilities to the Agent loop through tool schemas and approval checkpoints.

MCP is an integration and tool-provider boundary, not the Codex plugin system. The left navigation keeps `插件` as a separate later-stage plugin/skill surface, while MCP endpoint configuration stays in the settings view and MCP status/tool discovery is reachable through the local `/mcp` command.

The Agent loop follows the Codex reference in `codex-rs/core/src/session/turn.rs`: each sampling request may produce assistant text or tool calls; tool calls execute, their outputs are recorded, and the next sampling request continues until the model no longer needs follow-up, pending input takes over, user approval blocks, cancellation/error stops the turn, or context compaction must run. Godex therefore does not impose a fixed product-level step limit. `agent_loop_max_steps=0` means unbounded Codex-style follow-up; a positive value is only a diagnostic safety guard for tests or recovery, not the default execution contract. Automatic compaction, request approval, explicit cancellation, and transport failures are the real control boundaries.

Skill support is currently a local registry and prompt-injection boundary, not a full Codex marketplace. `GodexSkillRegistry` scans only local roots (`res://.agents/skills`, `res://.codex/skills`, `res://skills`, `res://addons/godex/skills`, and the matching user-profile `.agents`/`.codex` skill roots), treats `SKILL.md` as the skill entrypoint, and optionally reads `agents/openai.yaml` for interface, dependency, and policy metadata. The Settings Skill manager renders searchable local rows and path-level enable/disable buttons; disabled paths persist through `GodexState.skill_disabled_paths` into `user://godex/settings.json`. Registry models report `remote_enabled=false` and `marketplace_enabled=false`, so install/create/marketplace flows remain outside the current implementation.

The settings view is an application-level workspace, not a chat card. `SettingsPanel` swaps the center view into a Codex-like settings shell with a dedicated left settings rail, a `返回应用` row, stable category entries, and a centered scrollable settings content column. During settings mode, the normal app sidebar hosts the settings rail while `MainPanel` keeps the same rounded outer frame as chat mode, so switching between chat and settings feels seamless instead of changing to a different window shape. The settings content layout is `SettingsContentWrap(fill)` -> `SettingsScroll(fill)` -> `SettingsContentCenter(fill)` -> `SettingsContent(constrained width)`: the scroll region owns vertical movement, the center container owns horizontal centering, and legacy left/right spacer nodes stay hidden. The controller owns category filtering and search filtering so the rail behaves as navigation rather than decoration; unsupported entries such as appearance settings stay disabled until a real persistence-backed feature exists. Settings category rows use Godot editor-theme icons, muted disabled icon colors, and rounded selected backgrounds, while search with no matching sections shows `SettingsNoResults` instead of leaving a blank content column. Provider, API, MCP endpoint, permission, Skill, compression, and command-shell settings render as grouped cards with title/description copy on the left and controls on the right. Chat-only chrome such as the composer, floating right inspector, bottom drawer, change-review strip, and composer popovers must stay hidden while settings mode is active, so model/reasoning/context/approval menus never leak across the view boundary.

Provider settings treat `core/provider_catalog.gd` as the source of truth for preset API modes. OpenAI stays on the Responses API, while explicit OpenAI-compatible presets such as Yuren stay on Chat Completions and resolve to `/v1/chat/completions`. The settings controller blocks provider-field signals while synchronizing preset fields and forces Yuren back to Chat Completions if a stale API-mode widget still displays Responses API, preventing user-local provider selections from silently drifting to the wrong endpoint. Agent request boundaries also call `GodexState.normalize_runtime_provider()` before first-turn payloads, tool-result continuations, and retry dispatch, so an older in-memory session or retry snapshot cannot keep sending explicit Yuren requests to `/v1/responses`. Yuren remains a non-default preset, and personal API settings stay in `user://godex/settings.json` or environment variables rather than repository files.

OpenAI-compatible Chat Completions transport has a bounded degradation path. Explicit non-OpenAI Chat Completions providers start with Godot's direct non-stream `HTTPRequest` transport rather than the hand-polled stream client, because local probes showed the compatible provider can return valid non-stream text while live stream lifecycle handling is the more fragile boundary in the editor process. Provider identity is copied into each OpenAI request snapshot, so approval queues, retry requests, and tool-result continuations keep their original provider routing even if the user changes the settings UI before the request is sent. The first-send payload removes only `stream=true`; it preserves model, messages, tool schemas, and reasoning effort so the Agent still asks for the same capability shape. The completion handler updates the same assistant placeholder, stops the Agent loop with `final_model_response`, and records transport stage `non_stream_direct` with `stream=false`. If a compatible provider later rejects or drops richer request fields, Godex can start one plain Chat fallback that preserves the conversation messages but strips streaming, tool, parallel-call, and reasoning-only fields. If the failing payload is a tool-result continuation, the plain fallback converts `assistant.tool_calls` and `role=tool` messages into ordinary text summaries before retrying, because a tool-free compatibility request must not resend Chat tool protocol fields. Empty Chat completions are not treated as successful assistant answers: they either trigger the same plain fallback or become visible retryable failures with `empty_chat_completion` diagnostics. The text parser accepts both string `content` and common content-part arrays so compatible providers do not appear empty simply because they return structured text parts.

## Local Persistence

Godex stores user-local settings and conversation sessions under `user://godex`. Session persistence is intentionally JSON-based for now so the data is inspectable and easy to migrate while the Agent loop is still evolving. The current model supports active-thread restore, message history, search records, archive metadata, fork metadata, pin state, approval checkpoints, decision history, model events, local slash-command actions, active goals, sub-agent task records, sub-agent child-session links, session compaction, composer references, and local Skill disablement. Manual `/compact`, composer-menu compaction, and threshold-triggered automatic compaction all route through the same state boundary, store a latest compaction summary with before/after token estimates, retain a bounded compaction history, and publish `session_compaction` transcript/Automation evidence. `context_window_warning()` is the single state-owned threshold projection for capability previews and Automation rows, so UI code does not hard-code when a conversation is close to auto-compaction. Richer diff/review artifacts can be added as separate session fields later.

`GodexState.active_project` is the single project-name source for the left project rail and the empty-chat prompt. The controller refreshes it from Godot `ProjectSettings` on startup, and successful MCP project-state/tool responses may update it from `project_name` or a `project_path` fallback. UI code should not hard-code `Godot Project` for the hosted editor view.

The sidebar's primary top navigation is intentionally capped at four Codex app sections: new chat, search, plugins, and automation. These rows use the same rounded bordered hover/selected capsule contract as conversation rows, but they share one active surface with the thread list: when search/plugins/automation/settings is selected, active conversation rows clear their selected capsule; when chat is selected, top navigation selection clears and the active conversation row owns the highlight. Archived conversations are managed from Settings, not a persistent bottom-left chat sidebar button. Sidebar thread actions operate on explicit thread IDs. Each visible thread row is a full-width `PanelContainer` capsule: the active conversation keeps a rounded highlighted background, hovered rows lift to a subtle rounded background, hover state is reconciled from the actual pointer position so stale capsules clear when the pointer leaves, and the trailing age label is replaced only while hovered by inline `置顶` and `归档` actions. The archive action requires a second red `确认` click; leaving the row resets confirmation and restores the initial age-only state. Secondary actions such as rename/fork can still use `ThreadActionMenu`; rename opens a floating inline `ThreadRenamePanel` under the popover layer and commits through `GodexState.rename_session(thread_id, title)`, while pinning uses `GodexState.toggle_pin_session(thread_id)`. Row actions must not rely on whichever session happens to be active unless the action intentionally selects that row first. Pinned grouping is a render-only sidebar concern: `_rebuild_threads()` may insert non-interactive group labels, but `threads`, `to_model()`, search, and archived records remain plain session arrays.

Search results use the same explicit thread boundary: session matches render as `SearchResult_<thread_id>` buttons, while project/MCP/model reference rows remain static summaries. Opening a search result calls `select_thread(thread_id)`, saves the active selection, and returns to the chat view without changing archived-session filtering.

Archived conversations are intentionally separate from normal search and the active sidebar. `GodexState.archived_records(query)` lists only archived sessions with an optional archived-only filter, `restore_archived_session(thread_id)` clears the archived flag before selecting the session, and `delete_archived_session(thread_id)` removes only archived sessions. The UI renders these rows in the Settings-owned `ArchivedPanel` as `ArchivedResult_<thread_id>` rows with explicit delete and `取消归档` controls, so restore/delete behavior never depends on the active session.

## Local Slash Commands

Local slash commands cover lightweight workspace actions that should not require opening a settings page. `/mcp` opens the MCP status and tool-discovery view; it does not replace the settings fields that persist the MCP endpoint and enable/disable state.

The composer uses a compact Codex-style surface: a rounded outer panel, a dark multiline prompt field, and a fixed bottom control row for add-context, approval mode, goal, model, reasoning, IDE context, and one primary send/stop button. Idle clicks send immediately; running clicks stop the active request through the same primary button. Retry state belongs to diagnostics, queued intent, or future trace controls, not a second composer button. The add-context `+` control opens a scene-owned `AddContextPanel` under `ComposerPopoverLayer`. Its first-stage entries only execute existing safe boundaries: current-project MCP summary probe creation, optional recommended file context from the changed-file review summary, current-session compaction, state-backed Plan Mode toggling, goal tracking toggling, and plugin navigation. Project-summary and recommended-file actions record `context_menu_action` audit events, and recommended file rows additionally record `file_context` model events plus visible transcript messages. Plan Mode toggles record `plan_mode` events and persist with other agent settings. When `context_window_warning()` reports `warning` or `auto_ready`, the add-context button adopts a warning-tinted icon state, its tooltip explains the remaining distance to auto-compaction, the menu exposes the compaction action, and the send button tooltip previews that the next Agent turn may compact before request construction. Composer references now provide the first durable attachment boundary: selected assistant text can become removable reference chips above the prompt, reference-only sends are allowed, and sent user messages receive a `references` array that is consumed and cleared after submission. Supported image references are limited to URL or `data:image/` payload strings that the request builder can serialize as Responses `input_image` or Chat Completions `image_url` parts. They do not open a system file picker, read file contents, capture screenshots, or ingest arbitrary local image files yet; full local image input, editor screenshots, terminal/browser context, and plugin-provided sources remain future work.

Composer menus are reparented into a runtime `ComposerPopoverLayer` overlay and anchored to the prompt or the clicked pill. They deliberately do not participate in the `ComposerBox` layout, so opening the add-context menu, approval menu, slash-command menu, model picker, or reasoning picker cannot change the composer height. This mirrors Codex desktop's overlay behavior while keeping the menus as inspectable runtime controls for editor screenshots.

The right overlay is now a conversation inspector rather than a static progress demo. `ProgressOverlayLayer/RightRail` floats on the right side of the Godex main screen instead of living as a `Body` layout column, so the main conversation keeps its Codex-like reading width. It follows the Codex app task-sidebar boundary: progress, generated artifacts, sub-agents, and sources are separate concepts. `进度` is a model-controlled short-term plan/memory checklist rendered only from explicit `progress_items`; it must not be inferred from OpenAI request lifecycle, tool calls, command runs, retry state, queued-message drains, or Agent loop bookkeeping. OpenAI transport diagnostics, command timelines, and MCP execution state remain in model events, transcript disclosures, Automation, or the bottom terminal drawer unless the model explicitly publishes plan items. `输出` lists generated artifacts or changed files such as files from the change-review summary, not assistant prose, command stdout/stderr, MCP payloads, or model events. `来源` lists invoked external tool/context providers such as Godot .NET MCP, not internal event sources or local probe bookkeeping. New conversations start with `暂无产物` and `暂无来源`, while progress and sub-agent sections stay hidden until real events exist.

`GodexState.outputs` is the persisted artifact list for the right-inspector `输出` section. Change-review summaries publish each changed file as a `source=change_review` file artifact through the same state boundary, and clearing the review removes only those generated artifact rows. The controller may supplement legacy summaries while rendering, but it must deduplicate by source/path and must not infer artifacts from command logs, MCP payloads, local probe events, or assistant prose.

The header owns Codex-style layout chrome separately from functional navigation. `HeaderLayoutControls` contains a momentary launcher-menu button plus icon-only toggles for the bottom output drawer and floating right inspector. The launcher menu offers first-stage Codex-style entries for files, side chat, and terminal; it must not invent recommended document rows before a real source/recommendation pipeline exists, and it does not control Agent progress. These buttons are layout and launcher chrome only: they do not open MCP, Plugins, or Automation views. The right-inspector toggle does not hide the app navigation sidebar or change the fixed transcript width. MCP endpoint/status diagnostics are kept out of the visible top-right header and remain configured through settings and inspected through `/mcp`, while `插件` remains reserved for the later plugin/skill surface.

The chat view uses a fixed-width centered transcript column inside the available main area. When layout chrome changes, the controller animates the transcript column width with a short cubic tween so the conversation slides horizontally instead of snapping edge-to-edge. The outer main panel keeps a rounded Codex-like frame, while individual assistant, tool, and command rows remain transparent and borderless. Internal lifecycle/status records stay in model-event, Automation, progress, or terminal surfaces instead of becoming chat rows.

`BottomDrawer` gives the bottom layout toggle a real command terminal surface instead of a placeholder. It is a fixed-width, scrollable chat-view panel that rebuilds recent `command_run` records from session model events, showing command status, shell, working directory, runner metadata, exit/duration, bounded stdout/stderr or combined-output previews, stderr-merge notes, timeout-enforcement boundaries, and recent timeline transitions. It deliberately does not mirror right-inspector artifacts: `输出` remains generated files/artifacts, while the bottom terminal is command execution audit. The drawer stays hidden outside the chat view so search, settings, plugin, and automation panels do not get compressed by stale terminal content.

`ChangeReviewSurface` is the first-stage Codex-style changed-file review strip. It sits between the bottom drawer and `ComposerPanel`, outside `ComposerBox`, so it can appear above the composer without changing the prompt control height. `GodexState.change_review_summary` stores the auditable file count, addition/removal totals, cropped file rows, expanded state, and updated timestamp; the controller only renders this projection and does not read git or the filesystem. The strip shares the same fixed conversation-column width as `Messages`, `BottomDrawer`, and `ComposerPanel`, and its addition/removal labels have fixed right-aligned widths so `+9`, `+93`, and `+1000` do not move neighboring controls.

`GodexGitChangeSummaryService` is the read-only parser for future repository review data. It converts explicit `status --porcelain` / `diff --numstat` text or structured file rows into the same `change_review_summary` projection, normalizes paths for UI rows, and never starts a process or applies patches. The `godex_change_review_summary` model tool is therefore a local state-publication entry: it updates the review strip and records a completed tool call, but it does not create command-run events or approval checkpoints.

The approval menu uses a fixed-rect outer `Control` inside `ComposerPopoverLayer` and a themed inner `ApprovalModeSurface` for the Godot panel background. This keeps the visual popup short while preventing `PanelContainer` child minimum sizes from pushing the outer rect downward. Its height follows the Codex desktop permission list content and is clamped above the composer so it does not render a 525px blank lower area or overflow past the bottom edge.

Composer input that starts with a recognized slash command is handled locally by `GodexState.execute_slash_command()` and does not start an OpenAI request. Commands cover session creation, resume, fork, archive, rename, pin toggle, goal tracking, and manual compaction. `/goal` stores a session-level active goal record with objective, status, created/updated timestamps, usage placeholders, model-event audit records, and Automation summaries; active goals are injected into Agent instructions at request build time. The controller only displays the command result, saves sessions, and refreshes the active view, keeping command semantics out of the UI layer.

`GodexState.slash_command_suggestions()` is the slash-command discovery source for the composer. It returns filtered command metadata, aliases, parameter hints, action titles, muted descriptions, icon candidates, and insert text. The controller renders these suggestions inside the composer instead of hard-coding menu rows, so slash-command execution and slash-command discovery stay in the same state boundary.

The slash-command menu follows the Codex composer action-list pattern: a rounded dark popup above the input, one row per action, editor-theme icons on the left, a strong action title, muted description text, selected-row highlight, keyboard Up/Down selection, Enter insertion, Esc closing, and scrollable overflow. The titles and descriptions are display metadata only; command execution remains owned by `GodexState.execute_slash_command()`.

Unknown slash commands remain unhandled so future Agent or provider-specific command flows can decide whether to interpret them. This preserves normal prompt behavior while making Codex-style local workflow commands auditable and testable.

The composer send button now has a single primary behavior instead of opening a send-method popup. When the Agent is idle, it dispatches through the existing `_send_prompt()` path. When an Agent loop or OpenAI transport is running, it records the composer text through `GodexState.queue_user_message_with_action()`, clears the input, and renders queued rows above the composer. Queued rows expose per-message `引导` and delete actions: delete marks that queued record `cancelled`, while guide converts the queued text into a pending one-turn guide instruction and either sends immediately when idle or stays pending while the current turn is still running. Queued messages and guide instructions are session fields, model events, and Automation summaries, so they survive UI refresh and remain auditable without entering the chat transcript. Queued user messages drain one at a time through the same prompt-send boundary when the Agent loop is stopped, OpenAI credentials are available, and the composer has no unsent draft; successful final model responses schedule the next drain through a deferred lifecycle callback, and approval-mode requests preserve `queued_user_message` source metadata. Queued local slash commands are marked as `parse_slash` actions and drain through `_handle_slash_command()` without requiring an OpenAI API key, so `/goal`, `/compact`, and other local workflow commands keep their normal state effects while retaining queue submission attribution. Queued `!` prompts are the other OpenAI credential-gate exception: they are marked as `run_shell` queued actions and hand off to `command_run` plus command approval records, without invoking the local runner before approval. A guide instruction is consumed only at the next Agent request boundary: `GodexAgentService` injects it as user-supplied one-turn guidance, marks the pending steer as submitted, and does not treat it as a permanent system instruction or an approval-policy override. Multiple guide submissions update hidden pending-steer state and should not create repeated `已引导会话` transcript rows. True running-turn steer submission, rejected-steer priority, and review-mode steer rejection remain later execution-layer work.

## API Configuration Snapshot

Godex separates persisted provider settings from the runtime API request snapshot. `GodexState.api_config_snapshot()` resolves the active provider, model, reasoning effort, API mode, endpoint, key source, masked key display, and HTTP headers. Environment variables take priority over manually entered keys. The UI consumes only the masked status text, while request execution layers can consume generated headers without duplicating endpoint or authentication rules. `GodexProviderCatalog` keeps the Yuren OpenAI-compatible development preset in code, and `docs/references/yuren-openai-provider.json` preserves the source configuration for audit, but Yuren is not the default provider. Explicit Yuren settings and legacy `YURENAPI_API_KEY` values migrate on startup to `yurenapi` and `YUREN_API_KEY`; the recommended `https://yurenapi.cn/v1` endpoint is applied only when the existing Yuren base URL is empty or not in the allowed URL list. Existing `https://yurenapi.com/v1` settings are preserved as a valid persisted alternate. Yuren's catalog model list is intentionally limited to the supported OpenAI-style names `gpt-5.4-mini`, `gpt-5.4`, and `gpt-5.5`; settings and composer pickers read from this catalog instead of free-form defaults. Yuren and generic OpenAI-compatible presets default to `chat_completions`, so their runtime endpoint resolves to `/v1/chat/completions`; the default OpenAI preset remains on `responses`. Runtime settings are stored under Godot `user://godex/settings.json`; project-local copies such as `settings.json`, `godex/settings.json`, and `addons/godex/settings.json` are ignored to reduce accidental API setting leaks.

`tools/validate-godex.ps1` also treats settings leakage as a validation error. It rejects tracked local/secret JSON settings filenames and scans tracked text files for obvious raw OpenAI-style keys, bearer tokens, or inline JSON API key values. Provider examples must keep environment-variable placeholders such as `{env:YUREN_API_KEY}` rather than raw credentials.

## OpenAI Execution Boundary

`GodexOpenAIExecutionService` is the boundary between Agent planning and HTTP transport. It validates whether a request is ready to send, preserves masked authentication metadata for audits, exposes the selected reasoning effort, parses full Responses API or Chat Completions compatible output into text and tool-call records, and normalizes HTTP/API errors. Responses parsing accepts canonical `output[].content[]` text plus compatible provider convenience fields such as top-level `output_text`, `text`, and `refusal`; streaming parsing also treats `response.output_text.done`, `response.refusal.done`, and final `response.completed` payload text as terminal assistant output. It also normalizes SSE text deltas plus first-stage streamed tool-call fragments for Responses function-call events and Chat Completions `delta.tool_calls`. The request builder keeps one Responses-style tool schema at the Agent boundary and converts it into Chat Completions `function` wrappers only when building a chat-compatible payload. The editor UI owns transport lifecycle: `GodexDockController` uses `HTTPClient` for stream-enabled OpenAI requests, and direct `HTTPRequest` for explicit non-OpenAI Chat Completions first sends, while both paths update one assistant message in place, show a lightweight `正在思考` status with a left-to-right shimmer while work is in flight, hide successful OpenAI request/transport/response bookkeeping from the chat transcript, and route final bodies through the same execution service. The streaming path additionally accumulates streamed tool-call fragments, projects transient `正在解析` partial tool rows, and records final transcript tool rows when the stream completes. When a compatible provider disconnects with parseable trailing SSE data or returns a complete non-stream JSON body despite `stream=true`, the controller routes that residual buffer back through the same execution service before deciding whether the turn succeeded or failed; it also recognizes complete JSON bodies while the connection is still open so a non-standard provider does not look like a no-return conversation until idle timeout. Empty, partial, or invalid residual data remains a retryable stream failure. Non-2xx stream responses are provider diagnostics first: the controller reads a bounded response body preview, records the provider error type/message, and can retry once without `stream=true` when the status or body indicates stream/SSE compatibility trouble. If that non-stream Chat Completions fallback still fails on a compatibility-class status, Godex may make one final plain-chat fallback that recomputes the Chat Completions endpoint, strips `stream`, tool schemas, and reasoning-only fields from the same message payload, and audits the degraded request as `compatibility_fallback`. This path is limited to ordinary assistant text closure; it does not pretend tool execution succeeded. Bare credential-class failures such as a 401 are not retried without compatibility evidence. Transport events and retry snapshots carry a `stage` value (`provider_probe`, `stream`, `non_stream_direct`, `stream_fallback`, or `compatibility_fallback`) so the progress rail can label failures as provider probe, stream request, direct non-stream request, non-stream fallback, or plain compatibility fallback instead of collapsing them into a generic "OpenAI failed" state. Failure text remains visible through the assistant error message path; successful transport diagnostics remain in model events and progress/audit surfaces instead of being replayed as chat rows. Richer event replay should layer on top of this boundary instead of duplicating payload parsing in UI code.

When `GodexState.plan_mode_enabled` is active, `GodexAgentService` adds a Codex Plan Mode instruction contract to the initial OpenAI request and suppresses the tool schema list before the payload reaches the request builder. The corresponding audit snapshot exposes `plan_mode=true` and `tool_count=0`, while the chat transcript stays focused on user/assistant content rather than Plan Mode bookkeeping. This is an initial request-boundary implementation; richer Codex Plan Mode rendering, proposed-plan confirmation, and mode-exit workflows should build on the same state flag instead of reintroducing UI-only toggles.

When `GodexState.active_goal_record()` returns an enabled goal, `GodexAgentService` adds the objective and status to the OpenAI instructions and records the normalized goal in the turn audit. Paused, completed, and cleared goals remain visible as session state when appropriate, but only enabled statuses inject model context. Goal state is therefore a thread/session contract rather than a global composer boolean.

When `GodexState.skills_enabled` is active, `GodexAgentService` also reads the local registry projection through `enabled_skill_prompt_from_registry()`. That prompt advertises only enabled local Skills with `$skill` names, short descriptions, and implicit/explicit invocation policy; it does not blindly paste every `SKILL.md` body into normal turns. A user prompt that explicitly mentions `$name` triggers a narrow injection path: the service matches enabled registry records by name, reads the corresponding `SKILL.md`, strips frontmatter, appends the body under `Explicit Skill instructions selected by the user for this turn`, and records the injected names in the turn audit. Disabled Skills are ignored even when mentioned.

Sub-agent tasks are also session state. `GodexAgentService` asks `GodexSubagentManager` for a bounded task draft, then records it through `GodexState.record_subagent_task()` so the task has a stable ID, role, branch/read-only metadata, status, timestamps, summary, and optional result. Cancellation and result handoff are state transitions on the same record: `cancel_subagent_task()` marks queued/running/interrupted tasks as `cancelled` with source metadata, and `handoff_subagent_task_result()` records handoff summary/source metadata for finished tasks without inventing a second transcript object. Worker callbacks use `record_subagent_notification()` to store completed/failed/killed/running notices, patch the matching task, and create or update a `subagent_edges` parent-child record when a child thread ID is available. Edge records keep `parent_thread_id`, `child_thread_id`, optional task ID, open/closed status, timestamps, direct-child queries, and filtered descendant traversal so future worker sessions can restore their topology after editor rebuilds. `start_subagent_child_session()` adds the first local child-session boundary: it creates a child conversation with the prompt, records the parent task as running, links an open edge, and returns to the parent thread. `complete_subagent_child_session()` and `fail_subagent_child_session()` append the local child result/failure message and route the parent update through the same worker notification path. The right progress inspector prefers these session task records and falls back to legacy `subagent` model events for older sessions. Automation adds task, notification, and edge records as summary rows. This is still a local session/topology foundation rather than a real external sub-agent runner; future parallel execution should update these records through `update_subagent_task()`, `record_subagent_notification()`, and the lifecycle helpers instead of inventing a second UI data source.

Responses API tool-result continuations preserve provider response identity when it is available. Full and streamed response parsing attaches the response `id` to parsed function calls; the controller also caches response IDs from stream events such as `response.created` and backfills already accumulated tool-call fragments before they are recorded. `GodexState.record_tool_calls()` stores the ID on the pending tool-call event, and `GodexAgentService` passes it into the next Responses payload as `previous_response_id`. When that ID is present, the continuation sends only the `function_call_output` item for the original call ID. When the provider does not return a response ID, Godex keeps the older compatibility payload that includes the prior function call plus its output so non-storing OpenAI-compatible endpoints still receive enough context. OpenAI transport audit events retain safe continuation metadata (`source`, `tool_call_id`, `previous_response_id`, input count, and payload fingerprint) so a sent continuation can still be diagnosed after the pending continuation slot is cleared.

Local model replay uses `GodexAgentService.inject_model_response_replay()` to feed a deterministic Responses API-compatible fixture through the same `GodexOpenAIExecutionService` parser. Replay events are tagged with `source=local_model_replay`, `status=replayed`, and `fixture_name`; they create normal pending tool-call records for the MCP approval/execution path, but do not store or render transport headers or live request bodies. Replay diagnostics stay in model-event/Automation audit surfaces so they cannot be confused with an external OpenAI send, while the right-inspector `来源` chips stay limited to real invoked external tools.

Provider probing is a live transport diagnostic, not a replacement for the conversation loop. The Automation view exposes `Provider 探针`, which builds a minimal non-stream OpenAI-compatible request for the current API mode and model, sends it through a separate Godot `HTTPRequest` node from the editor process, and parses the response through the same execution service used by normal turns. Probe events are tagged with `source=provider_probe` so model-event/progress audit surfaces can show endpoint/model/status evidence, while `输出` remains reserved for generated artifacts and `来源` remains reserved for invoked external tools such as MCP or future web/search providers. OpenAI model transport, provider probes, local replay fixtures, file attachments, and project-summary context stay out of source chips. The probe never persists raw API keys and should be used to isolate Godot process networking, endpoint path, model compatibility, and parser behavior before debugging the larger streaming Agent loop.

Local tool-result continuation replay uses `GodexAgentService.replay_pending_tool_result_continuation()` to exercise the next Codex loop boundary without sending a network request. This is grounded in the local Codex reference implementation: `codex-rs/core/src/session/turn.rs` keeps sampling until function-call outputs are returned and a final assistant message arrives, while `codex-rs/core/src/tools/context.rs` and `protocol/src/models.rs` normalize MCP results into `FunctionCallOutput` items tied to the original call ID. Godex mirrors that contract by requiring a ready pending continuation payload, replaying a deterministic final Responses message through the same parser, tagging audit events with `source=local_tool_result_continuation_replay`, clearing the pending continuation slot, and stopping the Agent loop with `local_continuation_replay_final`. This proves the tool-result-to-final-message path is renderable and auditable; real provider behavior is still validated only by the normal OpenAI send path.

## Model and Reasoning Controls

The composer model and reasoning picker follows Codex desktop's combined menu shape: the composer footer exposes a single compact model/reasoning button, the left popover lists reasoning levels, and the bottom current-model row opens a right-side model submenu on hover. Leaving the combined hover region closes the model submenu instead of relying on click toggles. The model submenu renders the current `GodexProviderCatalog` model choices as compact checked rows and keeps catalog-external custom models visible instead of hiding the active setting. Selecting a row writes directly to `GodexState.model`; provider changes still rebuild the model list from `GodexProviderCatalog`. The settings page uses the same catalog for its default-model selector, and `GodexState.set_model()` normalizes invalid persisted names back to the selected provider's default before a request is built.

The reasoning side of the combined picker writes `GodexState.reasoning_effort` with `low`, `medium`, `high`, or `xhigh`. The picker is intentionally UI-only for now: provider/model metadata can later filter unsupported reasoning levels, but the existing request and audit pipeline already preserves the selected effort.

`GodexAgentService` passes the selected reasoning effort into all OpenAI payload builders. Responses API payloads use `reasoning.effort`, while Chat Completions-compatible payloads use `reasoning_effort`. Tool-result continuation requests retain the same effort so repeated Agent loop turns do not silently fall back to a different model behavior. Request snapshots and model events expose the effort for audit, while redacted headers continue to hide raw API keys.

## Transcript Tool Rows

The chat transcript now renders MCP tool execution and command requests as scene-owned Godot controls instead of plain assistant text. `GodexDockController` creates a `ToolTranscriptRow` when an MCP tool transport starts, updates the same row to `已运行` or `失败` when the tool result arrives, and keeps details collapsed by default behind a full-width disclosure header. Command requests use the parallel `CommandTranscriptRow`, with queued/approval-required/running/completed/failed/timed-out status text plus command, shell, working directory, runner kind, elapsed duration, exit code, combined output, stdout, stderr, stderr-merge handling, and timeout-enforcement details. User messages render as right-aligned `UserMessageBubble` controls with an expanding leading spacer and content-aware width limits, matching the Codex chat rhythm where user prompts sit apart from the assistant stream without short CJK prompts wrapping vertically. Assistant messages, tool rows, and command rows use transparent borderless containers so the transcript reads like Codex's continuous left-side conversation flow. `GodexState.active_transcript_items()` merges visible messages with real action rows: `tool_call`, `command_run`, and transient `partial_tool_call` previews. Agent loop, MCP context, queued-message, guide, Plan Mode, goal, compaction, sub-agent, OpenAI request/transport/response, and raw stream-trace diagnostics stay in model events, Automation, progress, or terminal surfaces instead of becoming chat rows. Tool and command transcript headers use semantic editor icons, visible hover/pressed highlights, and per-row expand state so real file/tool/terminal activity remains readable without reusing success checkmarks for hidden request bookkeeping. Partial tool-call rows are intentionally not persisted as session events and never enter `pending_tool_calls()`; they are only a live argument preview until the final model tool call is recorded.

Command execution is intentionally state-machine first. Godex exposes the Codex-compatible model tool name `exec_command` and keeps `godex_command_request` as a compatibility alias; both create an auditable command run rather than executing immediately. `exec_command` accepts Codex-shaped fields such as `command`, `workdir`, `timeout_ms`, `login`, `sandbox_permissions`, `justification`, and `prefix_rule`, but this first-stage bridge only maps command text, project-local working directory, and timeout into the existing approval-bound command-run model. `write_stdin` is also advertised with the Codex-compatible name, but it currently fails explicitly with `interactive_command_sessions_not_available` because a real process/session manager is not implemented yet. Enabling command capability allows command runs to enter an approval queue, not to execute. `GodexState.request_command_run_approval()` binds the command, normalized shell, project-local working directory, and timeout into a high-risk approval checkpoint. `execute_command_run_with_runner()` refuses blocked, disabled, unapproved, rejected, mutated, duplicate, concurrently running, or missing-runner commands, then accepts a supplied runner result and records completed/failed/timed-out metadata with bounded, redacted output. The UI now supplies a first-stage local runner for approved short commands: PowerShell and pwsh execute with `-NoLogo -NoProfile -NonInteractive -Command`, cmd executes with `/D /C`, and project-local `res://` working directories are entered inside the shell command instead of mutating the Godot editor process working directory. The current runner is synchronous and uses Godot `OS.execute`, so Godot merges stderr into the captured output when `read_stderr=true`; the result explicitly records `runner_kind=godot_os_execute_sync`, `duration_ms`, `combined_output`, `stderr_merged`, a stderr notice, and `timeout_enforced=false`. Long-running command sessions, process-tree termination, streaming stdout/stderr, hard timeout enforcement, `write_stdin`, `apply_patch`, and Codex's exec-server-style sandboxed filesystem backend remain later execution-layer work. `cancel_command_run()` marks queued, approval-required, approved, or running commands as cancelled and clears pending command approvals, but it still does not terminate an already blocking `OS.execute` call. The current working-directory policy is deliberately project-local: empty paths and `res://...` are allowed, while relative escape attempts, UNC paths, system folders, and user-home paths are blocked before approval. Future shell transport, live stdout/stderr streaming, command grouping, and change-delta badges should layer on this approval-bound state model instead of moving execution semantics into ad hoc UI code.

Automation surfaces command approvals separately from generic checkpoints. `pending_command_approval_summary()` shows the command ID, command text, shell, project-local working directory, timeout, and fingerprint before a user approves or rejects the request. `command_run_summary_rows()` lists recent command-run state without reusing empty OpenAI endpoint/model summary fields. The Automation approve/reject buttons still share the latest-checkpoint controls, but their tooltip targets the pending command ID and `decide_latest_approval()` routes command checkpoints through `decide_command_run_approval()`.

Automation command actions are deliberately separate from MCP tool execution. `RequestCommandApproval` advances the next queued command into an approval checkpoint without running anything. `ExecuteApprovedCommand` calls the approved-command execution boundary and passes the controller's local short-command runner; if a future controller lacks a runner, the same state boundary still records the `runner unavailable` failure path instead of invoking a shell. `CancelCommandRun` cancels the current queued, approval-required, approved, or running command through the same state boundary and clears its pending command approval when present. `ExecuteNextTool` remains MCP-only and must not consume command-run events.

Command transcript rows keep the Codex-style borderless disclosure row while exposing structured command output. The state projection provides both the legacy detail string and a structured `result` map for runner metadata, duration, exit code, combined output, stdout, stderr, streamed output chunks, and timeline rows; the controller renders those fields as compact sections inside `CommandTranscriptBody` without adding nested cards. Borderless transcript rows still carry transparent left/right content padding so chat text never touches the visual column edge. Transcript projection also re-applies command-output redaction and truncation so direct state updates cannot leak raw API keys into the UI. Command timelines are audit/view data attached to `command_run` events: they record queued, approval-required, approved, running, failed, timed-out, completed, blocked, and rejected transitions, but they do not start a process or alter approval fingerprints. Completed short-command timeline rows include elapsed duration when the runner reports it. Output chunks are only appended to an already-running command through `append_command_run_chunk()`, are bounded to avoid session growth, and are sanitized before storage, including API keys resolved from the configured environment variable.

## Conversation Layout Contract

The central chat surface uses a responsive capped Codex-style conversation column rather than a single hard-coded width. `Messages`, `ComposerPanel`, `BottomDrawer`, `ComposerQueueSurface`, and `ChangeReviewSurface` share one calculated target width. Wide editor layouts should approach the column maximum, while split or narrow layouts should shrink to the available center lane without falling back to an unrelated minimum or forcing horizontal overflow. Floating overlays such as the right progress rail do not reserve layout width or shrink the transcript; they may reduce the viewport-derived fallback only enough to avoid covering the working column. If the editor viewport is too narrow to show both the column and the right overlay with a safe gap, the right overlay hides before covering the transcript. Header buttons may show the launcher, bottom drawer, or sidebar, but they must not change the transcript width as a side effect. View transitions should animate dedicated surfaces in future passes, not by tweening the shared column width. The outer `MainPanel` owns the rounded chat-shell chrome, while transcript rows remain transparent and borderless with explicit side padding for readability. User message rows keep the same transcript width but place the bubble at the right edge through a spacer, so long assistant/tool rows and short user prompts do not compete for the same left-aligned visual lane.

## Model Events

Model events are session-scoped records that keep OpenAI request, response, tool-call, and error metadata separate from user-visible chat messages. `GodexState.append_model_event()` redacts bearer headers and drops raw response payloads before storing an event. The Automation view surfaces recent model event summaries so future streaming and retry flows can be audited without polluting the conversation transcript.

## Chat Display and Audit Data

The chat transcript should stay conversational. User-facing assistant replies render in full and are never folded by the chat renderer. Only tool and command transcript disclosures may collapse implementation details, with per-use headers, hover affordances, and a right-aligned chevron that remains visible while expanded. MCP tool results are projected as collapsible transcript rows instead of plain assistant summaries. Full MCP result data remains attached to tool-call model events and continuation payloads so Agent loop correctness and auditability are preserved without turning the chat column into a raw log viewer.

Transcript text should become selectable so users can copy with `Ctrl+C`. A non-destructive floating selection toolbar should offer `添加到对话` and `在侧边聊天中提问`; selected snippets become composer reference chips above the prompt, sharing the same future attachment model as images, files, screenshots, and other context sources. Image input is planned as the same reference-item pipeline: attached thumbnails appear above the composer, can be removed individually, and are serialized into the next model request only after the provider/request builder supports image parts.

## MCP Tool Dispatch

Parsed model tool calls are first stored as pending `tool_call` model events. `GodexAgentService.dispatch_tool_call()` applies the approval policy before producing a dispatch-ready MCP request. Low-risk calls can be auto-approved in assisted approval modes, while review mode records an approval checkpoint tied to the tool-call ID and waits for a user decision. Responses streaming treats `response.output_item.added` as partial tool metadata for the live preview only; final tool records are created from completed item payloads or a `response.completed` final output payload so half-built tool calls do not enter the pending queue. Dispatch readiness appends an `mcp_tool_dispatch` model event and updates the original tool call to `dispatch_ready`; future HTTP transport should consume this request and append tool-result events through the same model-event stream.

`GodexMcpClient` now builds JSON-RPC `tools/call` bodies and normalizes MCP tool results, including text content that contains serialized JSON payloads. `GodexAgentService.begin_tool_call_execution()` moves a dispatch-ready tool call into `executing`, and `handle_mcp_tool_call_response()` records a `mcp_tool_result` event before updating the original tool call to `succeeded` or `failed`. The Automation view owns the current small HTTPRequest transport and only forwards raw response text to the service parser.

For integration validation, `GodexAgentService.inject_mcp_context_probe()` can create a local `godex_mcp_context` tool call without waiting for an OpenAI response. It now opens a bounded local Agent loop slice, records a `local_tool_probe` model event, and creates the normal pending `tool_call` event. The Automation view exposes this as a probe action so the same UI can validate audit events, source chips for invoked MCP tooling, approval/dispatch behavior, MCP HTTP execution, and result recording against the live editor endpoint without inserting probe bookkeeping into chat.

Tool-call disclosure rows are intentionally summaries rather than raw transport dumps. Argument details are projected through a small safe-key whitelist and redaction helper, while result details prefer normalized message/error/summary text or field names. Full MCP result data remains in state for continuation construction, but the transcript, artifact list, and source chips should not render bearer headers, OpenAI payloads, raw MCP bodies, long tool arguments, or command output.

## Agent Loop Advancement

Godex's Agent loop is modeled after the local Codex source rather than a fixed iteration cap. The relevant Codex path is `codex/codex-rs/core/src/session/turn.rs`: a turn keeps following up while the model returns tool calls or pending input needs to be drained, and it stops on an actual end-turn, failure, or approval boundary rather than after an arbitrary count. Godex therefore uses `agent_loop_max_steps = 0` as the normal unbounded mode. A positive `agent_loop_max_steps` is only a diagnostic guard for tests or emergency troubleshooting.

The atomic loop shape is `正在思考 -> one model output item/batch -> tool call batch -> assistant continuation`. When a model response contains multiple tool calls, Godex treats them as one batch: it dispatches unresolved siblings through the same MCP approval/execution boundary, updates each call's state as it moves from running to terminal status, and projects the visible transcript as one expandable batch row for that turn. The expanded body lists the individual commands/tools. Only after the batch completes does Godex build the OpenAI tool-result continuation. It must not run a new model-generated batch of tools without giving the model a continuation opportunity to produce text. This mirrors Codex's app-server history model where command/MCP begin and end events are upserted into one turn item rather than appended as separate visible rows.

Pending user guidance follows Codex's pending-input boundary. The Codex reference keeps queued input in `TurnInputQueue` / `InputQueue` and drains it only when `can_drain_pending_input` is safe, before the next model request. Godex mirrors that with hidden pending-steer records: guide text sent during a running turn waits until the next request boundary, is injected as one-turn guidance, and is then marked submitted in place. It is not appended as a visible assistant status message and does not override approval policy.

The loop stops when a risky action requires approval, an OpenAI request is busy or canceled, credentials are missing, a continuation is ready but blocked by `请求批准` send review, a request fails, or the model returns final text with no pending input. The Automation view surfaces the current loop summary so users can see why Godex is still advancing or why it stopped.

## Codex-Derived Subagent Boundary

Codex subagents are independent child Agent threads, not ordinary chat messages. The local Codex reference records this in `codex/codex-rs/core/src/codex_delegate.rs`, `codex_thread.rs`, and the app protocol `Thread.ts`: delegated work is spawned with a parent thread id, `ThreadSource::Subagent`, and agent role/nickname metadata. Godex's current `subagent_tasks` state is therefore only the task/edge foundation. UI rows must describe autonomous child Agent work and handoff state, not pretend that a subagent is just another transcript paragraph.

Right-inspector subagent rows are filtered by that boundary. A record must either carry an explicit delegated source/kind or be backed by a child thread edge; historical context-probe or project-summary tasks without child-Agent evidence stay out of the `子智能体` rail even if they live in the same session state.

## Slash And Status Command Boundary

Slash commands are local workflow commands owned by `GodexState.execute_slash_command()`. The command palette should follow the Codex screenshot contract: rows expose icon, action title, and muted description for commands such as MCP status, personality, code review, side/branch chat, compression, feedback, model, reasoning mode, derived child work, and status. `/status` produces a compact session/context summary, while `/side` is the Codex-facing name for branching/forking the active conversation. Command output is state data and UI projection; unknown slash commands remain unhandled so future Agent or provider-specific command flows can decide how to interpret them.

## MCP Tool Discovery

`GodexMcpClient` can build a `tools/list` request and normalize JSON-RPC compatible responses into compact tool definitions. `GodexState.record_mcp_discovery()` caches the discovered tools, discovery status, error text, timestamp, and a model event, while `mcp_tool_summary_rows()` keeps the MCP management view scannable by summarizing only the tool name, description, and first few schema parameters. Actual HTTP transport should call the discovery parser and then feed results into this state boundary.

The current UI transport owns a small `HTTPRequest` node for `tools/list` only. It writes `request_ready`, `request_starting`, `request_sent`, transport error, HTTP error, and discovery result status back into `GodexState` instead of parsing responses in UI code. This keeps transport lifecycle concerns in the Control tree while preserving service-owned request construction and parsing.

## Approval Flow

Approval checkpoints are generated before risky planned actions such as MCP context injection, file writes, commands, runtime control, or network operations. A checkpoint records action, summary, risk, status, and decision time. The Automation view surfaces pending approvals and lets the user approve or reject the latest checkpoint before future execution layers consume the decision.

When `approval_mode` is `请求批准`, outbound OpenAI HTTP sends also enter the same approval checkpoint flow. First-turn user prompts, retry sends, and manual tool-result continuation sends record `network:openai_request` checkpoints with request source metadata, endpoint/model/body-size preview data, and in-memory transport data. UI summaries expose only redacted metadata; raw bearer headers remain in memory for the approved dispatch path and are not copied into model events or chat text.
